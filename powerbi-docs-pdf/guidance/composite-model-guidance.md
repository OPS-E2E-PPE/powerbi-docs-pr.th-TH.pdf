## <a name="optimize-model-design"></a><span data-ttu-id="dc455-127">ปรับแต่งการออกแบบแบบจำลอง</span><span class="sxs-lookup"><span data-stu-id="dc455-127">Optimize model design</span></span>

### <a name="table-storage-mode"></a><span data-ttu-id="dc455-129">โหมดตารางที่เก็บข้อมูล</span><span class="sxs-lookup"><span data-stu-id="dc455-129">Table storage mode</span></span>

<span data-ttu-id="dc455-130">ในแบบจำลองแบบรวม คุณสามารถกำหนดค่าโหมดที่เก็บข้อมูลสำหรับแต่ละตารางได้ (ยกเว้นตารางที่มีการคำนวณ):</span><span class="sxs-lookup"><span data-stu-id="dc455-130">In a Composite model, you can configure the storage mode for each table (except calculated tables):</span></span>

- <span data-ttu-id="dc455-142">**คิวรีตารางคู่หรือตาราง DirectQuery จากแหล่งข้อมูลเดียวกัน**: ข้อมูลทั้งหมดจะถูกดึงโดยการส่งคิวรีดั้งเดิมอย่างน้อยหนึ่งรายการไปยังแหล่งข้อมูล DirectQuery</span><span class="sxs-lookup"><span data-stu-id="dc455-142">**Queries Dual table(s) or DirectQuery table(s) from the same source**: All data is retrieved by sending one or more native queries to the DirectQuery source.</span></span> <span data-ttu-id="dc455-143">ซึ่งจะส่งผลให้มีประสิทธิภาพการทำงานที่เร็วที่สุดโดยเฉพาะอย่างยิ่งเมื่อมีดัชนีที่เหมาะสมในตารางต้นทาง</span><span class="sxs-lookup"><span data-stu-id="dc455-143">It will deliver the fastest possible performance, especially when appropriate indexes exist on the source tables.</span></span> <span data-ttu-id="dc455-144">สถานการณ์นี้เป็นเรื่องปกติสำหรับคิวรีที่เกี่ยวข้องกับตารางชนิดมีมิติแบบคู่และตารางชนิดข้อเท็จจริง DirectQuery</span><span class="sxs-lookup"><span data-stu-id="dc455-144">This scenario is common for queries that relate Dual dimension-type tables and DirectQuery fact-type tables.</span></span> <span data-ttu-id="dc455-145">คิวรีจะเป็น _แบบภายในกลุ่มแหล่งข้อมูล_ และความสัมพันธ์ทั้งหมดทั้งแบบหนึ่งต่อหนึ่งหรือหนึ่งต่อกลุ่มได้รับการประเมินเป็น[ความสัมพันธ์แบบปกติ](../transform-model/desktop-relationships-understand.md#regular-relationships)</span><span class="sxs-lookup"><span data-stu-id="dc455-145">These queries are _intra source group_, and so all one-to-one or one-to-many relationships are evaluated as [regular relationships](../transform-model/desktop-relationships-understand.md#regular-relationships).</span></span>
- <span data-ttu-id="dc455-146">**คิวรีอื่นๆ ทั้งหมด**: คิวรีเหล่านี้เกี่ยวข้องกับความสัมพันธ์แบบข้ามกลุ่มแหล่งข้อมูล</span><span class="sxs-lookup"><span data-stu-id="dc455-146">**All other queries**: These queries involve cross source group relationships.</span></span> <span data-ttu-id="dc455-147">เนื่องจากตารางนำเข้าเกี่ยวข้องกับตาราง DirectQuery หรือตารางแบบคู่เกี่ยวข้องกับตาราง DirectQuery จากแหล่งที่มาที่แตกต่างกัน ซึ่งในกรณีนี้จะทำงานเหมือนเป็นตารางนำเข้า</span><span class="sxs-lookup"><span data-stu-id="dc455-147">It's either because an Import table relates to a DirectQuery table, or a Dual table relates to a DirectQuery table from a different source—in which case it behaves as an Import table.</span></span> <span data-ttu-id="dc455-148">ความสัมพันธ์ทั้งหมดจะได้รับการประเมินเป็น [ความสัมพันธ์แบบจำกัด](../transform-model/desktop-relationships-understand.md#limited-relationships)</span><span class="sxs-lookup"><span data-stu-id="dc455-148">All relationships are evaluated as [limited relationships](../transform-model/desktop-relationships-understand.md#limited-relationships).</span></span> <span data-ttu-id="dc455-149">นอกจากนี้ยังหมายความว่าการจัดกลุ่มที่ใช้กับตารางที่ไม่ใช่ DirectQuery จะต้องถูกส่งไปยังแหล่งที่มา DirectQuery เป็นตารางเสมือน</span><span class="sxs-lookup"><span data-stu-id="dc455-149">It also means that groupings applied to non-DirectQuery tables must be sent to the DirectQuery source as a virtual table.</span></span> <span data-ttu-id="dc455-150">ในกรณีนี้ คิวรีดั้งเดิมอาจขาดประสิทธิภาพได้ โดยเฉพาะสำหรับชุดการจัดกลุ่มขนาดใหญ่</span><span class="sxs-lookup"><span data-stu-id="dc455-150">In this case, the native query can be inefficient, especially for large grouping sets.</span></span> <span data-ttu-id="dc455-151">และมีโอกาสที่จะเปิดเผยข้อมูลที่สำคัญในคิวรีดั้งเดิม</span><span class="sxs-lookup"><span data-stu-id="dc455-151">And, it has the potential to expose sensitive data in the native query.</span></span>

<span data-ttu-id="dc455-152">โดยสรุปแล้ว เราขอแนะนำให้คุณ:</span><span class="sxs-lookup"><span data-stu-id="dc455-152">In summary, we recommend that you:</span></span>

- <span data-ttu-id="dc455-153">พิจารณาอย่างรอบคอบว่าแบบจำลองแบบรวมคือโซลูชันที่เหมาะสม ในขณะที่อนุญาตให้มีการรวมในระดับแบบจำลองของแหล่งข้อมูลที่แตกต่างกัน นอกจากนี้ยังแนะนำความซับซ้อนของการออกแบบที่มีผลกระทบที่เป็นไปได้</span><span class="sxs-lookup"><span data-stu-id="dc455-153">Consider carefully that a Composite model is the right solution—while it allows model-level integration of different data sources, it also introduces design complexities with possible consequences</span></span>
- <span data-ttu-id="dc455-154">ตั้งค่าโหมดที่เก็บข้อมูลไปยัง **DirectQuery** เมื่อตารางที่เป็นตารางชนิดข้อเท็จจริงซึ่งจัดเก็บข้อมูลขนาดใหญ่ หรือจำเป็นต้องให้ข้อมูลใกล้เคียงกับการแสดงผลแบบเรียลไทม์</span><span class="sxs-lookup"><span data-stu-id="dc455-154">Set the storage mode to **DirectQuery** when a table is a fact-type table storing large data volumes, or it needs to deliver near real-time results</span></span>
- <span data-ttu-id="dc455-155">ตั้งค่าโหมดที่เก็บข้อมูลเป็น **คู่** เมื่อตารางเป็นตารางชนิดมีมิติและ จะได้รับการคิวรีร่วมกับ ตารางชนิดข้อเท็จจริง **DirectQuery** ที่ยึดตามแหล่งที่มาเดียวกัน</span><span class="sxs-lookup"><span data-stu-id="dc455-155">Set the storage mode to **Dual** when a table is a dimension-type table, and it will be queried together with **DirectQuery** fact-type tables based on the same source</span></span>
- <span data-ttu-id="dc455-156">กำหนดค่าความถี่ในการรีเฟรชที่เหมาะสมเพื่อให้แคชของแบบจำลองสำหรับตารางคู่ (และตารางที่มีการคำนวณที่ขึ้นต่อกัน) มีการซิงค์กับฐานข้อมูลต้นทาง</span><span class="sxs-lookup"><span data-stu-id="dc455-156">Configure appropriate refresh frequencies to keep the model cache for Dual tables (and any dependent calculated tables) in sync with the source database(s)</span></span>
- <span data-ttu-id="dc455-157">มุ่งมั่นที่จะทำให้แน่ใจว่าข้อมูลที่สอดคล้องกันในแหล่งข้อมูล (รวมถึงแคชแบบจำลอง) —ความสัมพันธ์แบบจำกัดจะนำแถวออก เมื่อค่าคอลัมน์ที่เกี่ยวข้องไม่ตรงกัน</span><span class="sxs-lookup"><span data-stu-id="dc455-157">Strive to ensure data integrity across data sources (including the model cache)—limited relationships will eliminate rows when related column values don't match</span></span>
- <span data-ttu-id="dc455-158">ปรับปรุงแหล่งข้อมูล DirectQuery ให้เหมาะสม ด้วยดัชนีที่เหมาะสมสำหรับการรวมที่มีประสิทธิภาพ การกรอง และการจัดกลุ่ม</span><span class="sxs-lookup"><span data-stu-id="dc455-158">Optimize DirectQuery data sources with appropriate indexes for efficient joins, filtering and grouping</span></span>
- <span data-ttu-id="dc455-159">อย่าโหลดข้อมูลที่สำคัญลงในตารางการนำเข้าหรือตารางคู่ ถ้ามีความเสี่ยงว่าคิวรีแบบดั้งเดิมจะถูกสกัดกั้น - สำหรับข้อมูลเพิ่มเติม ดู [ใช้แบบจำลองแบบรวมใน Power BI Desktop (ผลกระทบด้านความปลอดภัย)](../transform-model/desktop-composite-models.md#security-implications)</span><span class="sxs-lookup"><span data-stu-id="dc455-159">Don't load sensitive data into Import or Dual tables if there's risk of a native query being intercepted—for more information, see [Use composite models in Power BI Desktop (Security implications)](../transform-model/desktop-composite-models.md#security-implications)</span></span>

### <a name="aggregations"></a><span data-ttu-id="dc455-160">การรวม</span><span class="sxs-lookup"><span data-stu-id="dc455-160">Aggregations</span></span>

<span data-ttu-id="dc455-161">คุณสามารถเพิ่มการรวมไปยังตาราง DirectQuery ในแบบจำลองแบบรวมของคุณได้</span><span class="sxs-lookup"><span data-stu-id="dc455-161">You can add aggregations to DirectQuery tables in your Composite model.</span></span> <span data-ttu-id="dc455-162">การรวมจะถูกแคชไว้ในแบบจำลอ เพื่อให้ทำงานเป็นตารางนำเข้า (แม้ว่าจะไม่สามารถใช้งานได้เหมือนตารางแบบจำลอง)</span><span class="sxs-lookup"><span data-stu-id="dc455-162">Aggregations are cached in the model, and so they behave as Import tables (although they can't be used like a model table).</span></span> <span data-ttu-id="dc455-163">วัตถุประสงค์ คือ เพื่อปรับปรุงประสิทธิภาพการทำงานสำหรับคิวรีที่ "grain สูงขึ้น"</span><span class="sxs-lookup"><span data-stu-id="dc455-163">Their purpose is to improve performance for "higher grain" queries.</span></span> <span data-ttu-id="dc455-164">สำหรับข้อมูลเพิ่มเติม ดู[การรวมใน Power BI Desktop](../transform-model/desktop-aggregations.md)</span><span class="sxs-lookup"><span data-stu-id="dc455-164">For more information, see [Aggregations in Power BI Desktop](../transform-model/desktop-aggregations.md).</span></span>

<span data-ttu-id="dc455-165">เราแนะนำให้ตารางการรวมเป็นไปตามกฎพื้นฐาน: จำนวนแถวควรมีตัวคูณที่น้อยกว่าอย่างน้อย 10 ตัวคูณของตารางต้นแบบ</span><span class="sxs-lookup"><span data-stu-id="dc455-165">We recommend that an aggregation table follows a basic rule: Its row count should be at least a factor of 10 smaller than the underlying table.</span></span> <span data-ttu-id="dc455-166">ตัวอย่างเช่น ถ้าตารางต้นแบบจัดเก็บแถว 1000 ล้านแถว แล้วตารางการรวมไม่ควรเกิน 100 ล้านแถว</span><span class="sxs-lookup"><span data-stu-id="dc455-166">For example, if the underlying table stores 1 billion rows, then the aggregation table shouldn't exceed 100 million rows.</span></span> <span data-ttu-id="dc455-167">กฎนี้ช่วยให้แน่ใจว่ามีประสิทธิภาพการทำงานที่เพียงพอเทียบกับค่าใช้จ่ายในการสร้างและการรักษาตารางการรวม</span><span class="sxs-lookup"><span data-stu-id="dc455-167">This rule ensures there's an adequate performance gain relative to the cost of creating and maintaining the aggregation table.</span></span>
