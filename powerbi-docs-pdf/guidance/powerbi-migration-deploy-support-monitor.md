# <a name="deploy-to-power-bi"></a>ปรับใช้ไปยัง Power BI

บทความนี้จะอธิบายถึง **ขั้นตอนที่ 5** ซึ่งเกี่ยวข้องกับการปรับใช้การสนับสนุนและการตรวจสอบเนื้อหาเมื่อโยกย้ายข้อมูลไปยัง Power BI

:::image type="content" source="media/powerbi-migration-deploy-support-monitor/migrate-to-powerbi-stage-5.png" alt-text="รูปภาพจะถึงแสดงขั้นตอนของการโยกย้ายข้อมูลไปยัง Power BI ขั้นตอนที่ 5 จะถูกเน้นสำหรับบทความนี้":::

> [!NOTE]
> สำหรับคำอธิบายที่สมบูรณ์ของกราฟิกด้านบน โปรดดูที่[ภาพรวมการโยกย้ายข้อมูล Power BI](powerbi-migration-overview.md)

จุดมุ่งเน้นหลักของขั้นตอนที่ 5 คือการปรับใช้โซลูชัน Power BI ใหม่ในการผลิต

ผลลัพธ์จากขั้นตอนนี้เป็นโซลูชันการผลิตที่พร้อมสำหรับการใช้งานโดยธุรกิจ เมื่อทำงานกับวิธีการแบบว่องไว ซึ่งเป็นวิธีที่่ยอมรับได้ที่จะมีการปรับปรุงที่วางแผนไว้ซึ่งจะส่งมอบในการทำซ้ำในอนาคต การสนับสนุนและการตรวจสอบก็มีความสำคัญเช่นกันในขั้นตอนนี้และมีความสำคัญอย่างต่อเนื่อง

> [!TIP]
> ยกเว้นการทำงานแบบขนานและการยกเลิกการใช้งานรายงานเดิม ซึ่งจะกล่าวถึงด้านล่าง หัวข้อที่กล่าวถึงในบทความนี้ยังใช้กับโครงการการใช้งาน Power BI แบบมาตรฐาน

## <a name="deploy-to-test-environment"></a>ปรับใช้เพื่อทดสอบสภาพแวดล้อม

สำหรับโซลูชันที่จัดการด้วยไอที หรือโซลูชันที่มีความสำคัญต่อประสิทธิผลทางธุรกิจโดยทั่วไปจะมีการทดสอบสภาพแวดล้อม การทดสอบสภาพแวดล้อมอยู่ระหว่างการพัฒนาและการผลิตและไม่จำเป็นสำหรับโซลูชัน Power BI ทั้งหมด การทดสอบพื้นที่ทำงานสามารถใช้เป็นตำแหน่งที่ตั้งที่มั่นคงได้ ซึ่งแยกออกจากการพัฒนาเพื่อให้การทดสอบการยอมรับของผู้ใช้ (UAT) เกิดขึ้นก่อนที่จะเผยแพร่สู่การใช้งานจริง

หากเนื้อหาของคุณได้รับการเผยแพร่ไปยังพื้นที่ทำงานบนความจุระดับพรีเมียม [การปรับใช้ไปป์ไลน์](../create-reports/deployment-pipelines-overview.md)สามารถทำให้ขั้นตอนการปรับใช้ง่ายขึ้นเพื่อพัฒนาทดสอบและพื้นที่ทำงานการผลิต อีกทางหนึ่ง การเผยแพร่อาจทำได้ด้วยตนเองหรือด้วย[สคริปต์ PowerShell](https://powerbi.microsoft.com/blog/duplicating-workspaces-by-using-power-bi-cmdlets/)ได้

### <a name="deploy-to-test-workspace"></a>ปรับใช้เพื่อทดสอบพื้นที่ทำงาน

กิจกรรมหลักระหว่างการปรับใช้กับการทดสอบพื้นที่ทำงานโดยทั่วไป ได้แก่:

- **สตริงและพารามิเตอร์ที่เชื่อมต่อ:** ปรับสตริงการเชื่อมต่อชุดข้อมูล หากแหล่งข้อมูลแตกต่างกันระหว่างการพัฒนาและการทดสอบ [การกำหนดพารามิเตอร์](../connect-data/service-parameters.md)สามารถใช้เพื่อจัดการสตริงการเชื่อมต่อได้อย่างมีประสิทธิภาพ
- **เนื้อหาพื้นที่ทำงาน:** เผยแพร่ชุดข้อมูลและรายงานไปยังพื้นที่ทำงานทดสอบและสร้างแดชบอร์ด
- **แอป** เผยแพร่[แอป](../consumer/end-user-apps.md)โดยใช้เนื้อหาจากพื้นที่ทำงานทดสอบหากจะเป็นส่วนหนึ่งของกระบวนการ UAT โดยปกติแล้ว การอนุญาตของแอพจะจำกัดเฉพาะผู้ที่เกี่ยวข้องกับ UAT เพียงจำนวนเล็กน้อย
- **การรีเฟรชข้อมูล:** [กำหนดตารางเวลาการรีเฟรช](../connect-data/refresh-scheduled-refresh.md)สำหรับชุดข้อมูลนำเข้าใด ๆ สำหรับช่วงเวลาที่ UAT กำลังเกิดขึ้น
- **ความปลอดภัย:** อัปเดตหรือตรวจสอบ [บทบาทพื้นที่ทำงาน](../collaborate-share/service-new-workspaces.md#roles-in-the-new-workspaces) การทดสอบการเข้าถึงพื้นที่ทำงานรวมถึงคนจำนวนน้อยที่เกี่ยวข้องกับ UAT

> [!NOTE]
> สำหรับข้อมูลเพิ่มเติมเกี่ยวกับตัวเลือกสำหรับการปรับใช้เพื่อพัฒนาทดสอบและใช้งานจริง โปรดดูส่วนที่ 9 ของ [การวางแผนเอกสารทางเทคนิคการปรับใช้ Power BI สำหรับองค์กร](https://aka.ms/PBIEnterpriseDeploymentWP)

### <a name="conduct-user-acceptance-testing"></a>ดำเนินการทดสอบการยอมรับของผู้ใช้

โดยทั่วไปแล้ว UAT จะเกี่ยวข้องกับผู้ใช้ทางธุรกิจที่เป็นผู้เชี่ยวชาญเฉพาะเรื่อง เมื่อได้รับการตรวจสอบแล้ว พวกเขาจะให้การรับรองว่าเนื้อหาใหม่ถูกต้อง ตรงตามข้อกำหนด และอาจนำไปใช้เพื่อการบริโภคในวงกว้างโดยผู้อื่น

ขอบเขตที่กระบวนการซึ่ง UAT นี้เป็นทางการ รวมถึงการลงนามเป็นลายลักษณ์อักษรจะขึ้นอยู่กับแนวทางปฏิบัติในการจัดการการเปลี่ยนแปลงของคุณ

## <a name="deploy-to-production-environment"></a>ปรับใช้กับสภาพแวดล้อมการผลิต

มีข้อควรพิจารณาหลายประการสำหรับการปรับใช้กับสภาพแวดล้อมการผลิต

### <a name="conduct-a-staged-deployment"></a>ดำเนินการปรับใช้แบบทีละขั้นตอน

หากคุณกำลังพยายามลดความเสี่ยงและการหยุดชะงักของผู้ใช้ หรือหากมีข้อกังวลอื่น ๆ คุณอาจเลือกที่จะปรับใช้แบบทีละขั้นตอน การนำไปใช้งานจริงครั้งแรกอาจเกี่ยวข้องกับผู้ใช้นำร่องกลุ่มเล็ก ๆ ด้วยการนำร่อง สามารถขอคำติชมจากผู้ใช้นำร่องได้

ขยายสิทธิ์ในพื้นที่ทำงานการผลิต หรือแอป ทีละน้อยจนกว่าผู้ใช้เป้าหมายทั้งหมดจะมีสิทธิ์ในโซลูชัน Power BI ใหม่

> [!TIP]
> ใช้[บันทึกกิจกรรม Power BI](../admin/service-admin-auditing.md)เพื่อทำความเข้าใจว่าผู้บริโภคใช้และใช้โซลูชัน Power BI ใหมกัน่อย่างไร

### <a name="handle-additional-components"></a>จัดการส่วนประกอบเพิ่มเติม

ในระหว่างขั้นตอนการปรับใช้คุณอาจต้องทำงานร่วมกับผู้ดูแลระบบ Power BI ของคุณเพื่อจัดการกับข้อกำหนดอื่น ๆ ที่จำเป็นในการสนับสนุนโซลูชันทั้งหมด เช่น:

- **การบำรุงรักษาเกตเวย์:** [แหล่งข้อมูลใหม่](../connect-data/service-gateway-data-sources.md)อาจจำเป็นต้องลงทะเบียนใหม่ในเกจเวย์ข้อมูล
- **ไดรเวอร์และตัวเชื่อมต่อเกตเวย์:** แหล่งข้อมูลที่เป็นกรรมสิทธิ์ใหม่อาจต้องมีการติดตั้งไดรเวอร์ใหม่หรือตัวเชื่อมต่อแบบกำหนดเองบนเซิร์ฟเวอร์แต่ละตัวในคลัสเตอร์เกตเวย์
- **สร้างความจุแบบพรีเมียมขึ้นใหม่:** คุณอาจสามารถใช้ [ความจุแบบพรีเมียม](../admin/service-premium-capacity-manage.md)ที่มีอยู่ได้ หรืออาจมีสถานการณ์ที่รับประกันความจุแบบพรีเมียมใหม่ได้ ซึ่งอาจเป็นกรณีที่คุณตั้งใจจะแยกภาระงานของแผนกออกจากกัน
- **ตั้งค่ากระแสข้อมูล Power BI:** กิจกรรมการเตรียมข้อมูลสามารถตั้งค่าได้ครั้งเดียวใน [กระแสข้อมูล Power BI](../transform-model/dataflows/dataflows-introduction-self-service.md) โดยใช้ Power Query Online ซึ่งช่วยหลีกเลี่ยงการจำลองงานการเตรียมข้อมูลในไฟล์ Power BI Desktop ต่าง ๆ ได้
- **ลงทะเบียนภาพขององค์กรใหม่:** การลงทะเบียน [ภาพขององค์กร](../developer/visuals/power-bi-custom-visuals-organization.md)สามารถทำได้ในพอร์ทัลผู้ดูแลระบบสำหรับภาพแบบกำหนดเองที่ไม่ได้มาจาก AppSource
- **กำหนดเนื้อหาเด่น:** มีการตั้งค่าผู้เช่าที่ควบคุมว่าใครอาจ [แสดงเนื้อหา](https://powerbi.microsoft.com/blog/promote-your-reports-dashboards-and-apps-on-power-bi-home/)ในโฮมเพจของบริการ Power BI
- **ตั้งค่าป้ายชื่อความลับ:** [ป้ายชื่อความลับ](../admin/service-security-data-protection-overview.md)ทั้งหมดถูกรวมเข้ากับ Microsoft Information Protection

### <a name="deploy-to-production-workspace"></a>ปรับใช้กับพื้นที่ทำงานการผลิต

กิจกรรมหลักระหว่างการปรับใช้กับพื้นที่ทำงานการผลิตโดยทั่วไป ได้แก่ :

- **การจัดการการเปลี่ยนแปลง:** ในกรณีที่จำเป็น จะขอรับการอนุมัติในการปรับใช ้และสื่อสารการปรับใช้กับประชากรผู้ใช้โดยใช้แนวทางปฏิบัติในการจัดการการเปลี่ยนแปลงมาตรฐานของคุณ อาจมีหน้าต่างการจัดการการเปลี่ยนแปลงที่ได้รับอนุมัติในระหว่างที่อนุญาตให้ใช้งานการผลิตได้ โดยปกติแล้ว จะใช้ได้กับเนื้อหาที่มีการจัดการด้วยไอทีและมักจะใช้กับเนื้อหาแบบบริการตนเองน้อยกว่ามาก
- **การวางแผนแบบย้อนกลับ:** ด้วยการโยกย้ายข้อมูล ความคาดหวังก็คือการโยกย้ายโซลูชันใหม่เป็นครั้งแรก หากมีเนื้อหาอยู่แล้ว ควรมีแผนที่จะเปลี่ยนกลับไปใช้เวอร์ชันก่อนหน้าหากมีความจำเป็น มีไฟล์ Power BI Desktop เวอร์ชันก่อนหน้า (โดยใช้ SharePoint หรือ OneDrive ที่กำหนดรุ่น) จะทำงานได้ดีสำหรับวัตถุประสงค์นี้
- **สตริงและพารามิเตอร์ที่เชื่อมต่อ:** ปรับสตริงการเชื่อมต่อชุดข้อมูลเมื่อแหล่งข้อมูลมีความแตกต่างกันระหว่างการทดสอบและการผลิต [การกำหนดพารามิเตอร์](../connect-data/service-parameters.md)สามารถใช้งานได้อย่างมีประสิทธิภาพเพื่อการนี้
- **การรีเฟรชข้อมูล:** [กำหนดตารางเวลาการรีเฟรชชุดข้อมูล](../connect-data/refresh-scheduled-refresh.md)สำหรับชุดข้อมูลใด ๆ ที่นำเข้ามา
- **เนื้อหาพื้นที่ทำงาน:** เผยแพร่ชุดข้อมูลและรายงานไปยังพื้นที่ทำงานการผลิต และสร้างแดชบอร์ด [การปรับใช้ไปป์ไลน์](../create-reports/deployment-pipelines-overview.md)สามารถลดความซับซ้อนของกระบวนการเพื่อปรับใช้กับพื้นที่ทำงานการพัฒนาทดสอบและการผลิต หากเนื้อหาของคุณได้รับการเผยแพร่ไปยังพื้นที่ทำงานบนความจุแบบพรีเมียมได้
- **แอป:** หากแอปเป็นส่วนหนึ่งของกลยุทธ์การกระจายเนื้อหาของคุณ เผยแพร่ [แอป](../consumer/end-user-apps.md)โดยใช้เนื้อหาจากพื้นที่ทำงานการผลิต
- **ความปลอดภัย:** อัปเดทและตรวจสอบ [บทบาทพื้นที่ทำงาน](../collaborate-share/service-new-workspaces.md#roles-in-the-new-workspaces)ตามการกระจายเนื้อหาและกลยุทธ์การทำงานร่วมกันของคุณ
- **การตั้งค่าชุดข้อมูล:** อัปเดตและตรวจสอบการตั้งค่าสำหรับชุดข้อมูลแต่ละชุด ประกอบไปด้วย:
  - [การรับรอง](../collaborate-share/service-endorse-content.md) (เช่น ได้รับการรับรองหรือเลื่อนตำแหน่ง)
  - การเชื่อมต่อเกตเวย์หรือข้อมูลรับรองแหล่งข้อมูล
  - รีเฟรชตามกำหนดการ
  - [คำถามถาม & ตอบที่น่าสนใจ](../create-reports/service-q-and-a-create-featured-questions.md)
- **การตั้งค่ารายงานและแดชบอร์ด:** อัปเดตและตรวจสอบการตั้งค่าสำหรับแต่ละรายงานและแดชบอร์ด การตั้งค่าที่สำคัญที่สุด ได้แก่:
  - คำอธิบาย
  - บุคคลหรือกลุ่มที่ติดต่อ
  - [ป้ายชื่อความลับ](../admin/service-security-apply-data-sensitivity-labels.md)
  - [เนื้อหาที่โดดเด่น](https://powerbi.microsoft.com/blog/promote-your-reports-dashboards-and-apps-on-power-bi-home/)
- **การสมัครใช้งาน:** ตั้งค่าการสมัครสมาชิกรายงานหากมีความจำเป็น

> [!IMPORTANT]
> ณ จุดนี้คุณได้ก้าวสู่ความสำเร็จครั้งยิ่งใหญ่ เฉลองความสำเร็จของคุณในการโยกย้ายข้อมูลได้สำเร็จเสร็จสิ้น

### <a name="communicate-with-users"></a>สื่อสารกับผู้ใช้งาน

ประกาศโซลูชันใหม่ให้กับผู้บริโภค แจ้งให้พวกเขาทราบว่าสามารถค้นหาเนื้อหาได้จากที่ใด รวมถึงเอกสารประกอบ คำถามที่พบบ่อย และคำแนะนำ หากต้องการแนะนำเนื้อหาใหม่ให้พิจารณาจัดเซสชันประเภทเปิดใช้และเรียนรู้หรือเตรียมวิดีโอตามคำขอ

ตรวจสอบให้แน่ใจว่าใส่คำแนะนำเกี่ยวกับวิธีขอความช่วยเหลือ ตลอดจนวิธีการให้ข้อเสนอแนะแล้ว

### <a name="conduct-a-retrospective"></a>ดำเนินการย้อนหลัง

พิจารณาการดำเนินการย้อนหลังเพื่อตรวจสอบว่าอะไรเป็นไปได้ดีกับการโยกย้ายข้อมูลและสิ่งที่สามารถทำได้ดีขึ้นในการโยกย้ายข้อมูลครั้งต่อไป

## <a name="run-in-parallel"></a>ใช้งานแบบคู่ขนาน

ในหลายสถานการณโซลูชันใหม่จะทำงานควบคู่ไปกับโซลูชันเดิมตามเวลาที่กำหนดไว้ล่วงหน้า ข้อดีของการใช้งานแบบคู่ขนาน ประกอบไปด้วย:

- ลดความเสี่ยงลงโดยเฉพาะอย่างยิ่งหากการรายงานถือเป็นภารกิจสำคัญ
- ให้เวลาสำหรับผู้ใช้เพื่อให้คุ้นเคยกับโซลูชัน Power BI ใหม่
- อนุญาตให้ข้อมูลที่นำเสนอใน Power BI สามารถอ้างอิงข้ามกับรายงานเดิมได้

## <a name="decommission-the-legacy-report"></a>ยกเลิกรายงานเดิม

ในบางจุดรายงานที่ถูกโยกย้ายข้อมูลไปยัง Power BI ควรมีการปิดใช้งานในแพลตฟอร์ม BI เดิม ยกเลิกการใช้รายงานเดิมอาจเกิดขึ้นได้เมื่อ:

- เวลาที่กำหนดไว้ล่วงหน้าสำหรับการทำงานแบบคู่ขนาน—ซึ่งควรได้รับการสื่อสารกับประชากรผู้ใช้—ได้หมดอายุลงแล้ว โดยทั่วไปแล้วจะอยู่ที่ 30 ถึง 90 วัน
- ผู้ใช้ทั้งหมดของระบบเดิมมีสิทธิ์ในการเข้าถึงโซลูชัน Power BI ใหม่
- กิจกรรมที่สำคัญจะไม่เกิดขึ้นในรายงานเดิมอีกต่อไป
- ไม่มีปัญหาใด ๆ เกิดขึ้นกับโซลูชัน Power BI ใหม่ ที่อาจส่งผลกระทบต่อประสิทธิภาพการทำงานของผู้ใช้

## <a name="monitor-the-solution"></a>ตรวจสอบโซลูชัน

กิจกรรมจาก[การบันทึกกิจกรรม Power BI](../admin/service-admin-auditing.md)สามารถใช้เพื่อทำความเข้าใจรูปแบบการใช้งานของโซลูชันใหม่่ (หรือ[บันทึกการดำเนินการ](/sql/reporting-services/report-server/report-server-executionlog-and-the-executionlog3-view)สำหรับเนื้อหาที่ปรับใช้กับเซิร์ฟเวอร์การรายงาน Power BI) การวิเคราะห์บันทึกกิจกรรมสามารถช่วยตรวจสอบว่าการใช้งานจริงแตกต่างไปจากความคาดหวังหรือไม่ นอกจากนี้ยังสามารถตรวจสอบว่าโซลูชันได้รับการสนับสนุนอย่างเพียงพอได้หรือไม่อีกด้วย

นี่คือคำถามบางส่วนที่สามารถแก้ไขได้โดยการตรวจสอบบันทึกกิจกรรม:

- มีการดูเนื้อหาบ่อยเพียงใด?
- ใครกำลังดูเนื้อหาอยู่?
- โดยทั่วไปเนื้อหาจะดูผ่านทางแอปหรือพื้นที่ทำงาน?
- ผู้ใช้ส่วนใหญ่จะใช้ผ่านเบราว์เซอร์หรือแอปพลิเคชันมือถือ?
- มีการใช้งานการสมัครสมาชิกหรือไม่?
- มีการสร้างรายงานใหม่โดยใช้ตามโซลูชันนี้หรือไม่?
- มีการอัปเดตเนื้อหาบ่อยเพียงใด?
- ความปลอดภัยถูกกำหนดไว้อย่างไร?
- มีปัญหาที่เกิดขึ้นเป็นประจำ เช่น การรีเฟรชข้อมูลล้มเหลวหรือไม่?
- กิจกรรมที่เกิดขึ้น (ตัวอย่างเช่น กิจกรรมการส่งออกที่สำคัญหรือการแบ่งปันรายงานส่วนบุคคลจำนวนมาก) ซึ่งอาจหมายถึงการฝึกอบรมเพิ่มเติมอาจได้รับการรับประกันหรือไม่?

> [!IMPORTANT]
> ตรวจสอบให้แน่ใจว่ามีคนตรวจสอบบันทึกกิจกรรมอยู่เป็นประจำ การบันทึกและจัดเก็บประวัติเพียงอย่างเดียวมีประโยชน์ต่อวัตถุประสงค์ในการตรวจสอบหรือการปฏิบัติตามข้อกำหนด อย่างไรก็ตามค่าที่แท้จริงคือเมื่อสามารถดำเนินการเชิงรุกได้

## <a name="support-the-solution"></a>การสนับสนุนโซลูชัน

แม้ว่าการโยกย้ายข้อมูลจะเสร็จสมบูรณ์แล้ว แต่ช่วงเวลาหลังการโยกย้ายข้อมูลมีความสำคัญต่อการแก้ไขปัญหาและจัดการข้อกังวลด้านประสิทธิภาพใด ๆ เมื่อเวลาผ่านไปโซลูชันที่โยกย้ายข้อมูลแล้วมีแนวโน้มที่จะเปลี่ยนแปลงไปตามความต้องการทางธุรกิจ

การสนับสนุนมีแนวโน้มที่จะเกิดขึ้นแตกต่างกันเล็กน้อยขึ้นอยู่กับการจัดการ BI แบบบริการตนเองทั่วทั้งองค์กร ตัวแทนของ Power BI ทั่วทั้งหน่วยธุรกิจมักทำหน้าที่เป็นฝ่ายสนับสนุนบรรทัดแรกอย่างไม่เป็นทางการ แม้ว่าจะเป็นบทบาทที่ไม่เป็นทางการ แต่ก็เป็นบทบาทสำคัญที่ควรได้รับการสนับสนุน

การมีกระบวนการสนับสนุนอย่างเป็นทางการซึ่งมีเจ้าหน้าที่ฝ่ายไอทีพร้อมตั๋วสนับสนุน เป็นสิ่งจำเป็นสำหรับการจัดการคำขอที่มุ่งเน้นระบบตามปกติและเพื่อวัตถุประสงค์ในการยกระดับอีกด้วย

นอกจากนี้คุณยังอาจมี [ศูนย์แห่งความเป็นเลิศ (COE)](center-of-excellence-establish.md)ที่ทำหน้าที่เหมือนที่ปรึกษาภายในที่สนับสนุนให้ความรู้และควบคุม Power BI ในองค์กร COE สามารถรับผิดชอบในการดูแลจัดการเนื้อหา Power BI ที่เป็นประโยชน์ในพอร์ทัลภายใน

สุดท้ายนี้ ผู้บริโภคเนื้อหาควรทราบอย่างชัดเจนว่าควรติดต่อใครเมื่อมีคำถามเกี่ยวกับเนื้อหาและมีกลไกในการให้ข้อเสนอแนะเกี่ยวกับปัญหาหรือการปรับปรุง
