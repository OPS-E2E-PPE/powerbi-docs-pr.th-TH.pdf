# <a name="learn-from-customer-power-bi-migrations"></a>เรียนรู้จากการย้ายข้อมูล Power BI ของลูกค้า

บทความนี้ ซึ่งจะสรุปชุดข้อมูลเกี่ยวกับการโยกย้ายข้อมูลไปยัง Power BI แชร์คีย์บทเรียนที่ได้เรียนรู้จากลูกค้าสองรายที่โยกย้ายข้อมูลไปยัง Power BI สำเร็จแล้ว

## <a name="international-consumer-goods-company"></a>บริษัทสินค้าอุปโภคบริโภคระหว่างประเทศ

บริษัทสินค้าอุปโภคบริโภคระหว่างประเทศ ซึ่งขายผลิตภัณฑ์หลายร้อยรายการตัดสินใจที่จะใช้กลยุทธ์ระบบคลาวด์ในปี 2017 ปัจจัยที่สำคัญประการหนึ่งในการเลือก Power BI เป็นแพลตฟอร์ม Business Intelligence (BI) คือการผสานรวมกับ Azure และ Microsoft 365 ในเชิงลึก

### <a name="conduct-a-phased-migration"></a>ดำเนินการโยกย้ายข้อมูลเป็นระยะ

บริษัทเริ่มใช้ Power BI ในปี 2017 วัตถุประสงค์แรกเริ่มขององค์กรคือการแนะนำ Power BI เป็นเครื่องมือ BI เพิ่มเติม การตัดสินใจดังกล่าวทำให้ผู้เขียนเนื้อหา ผู้บริโภค และเจ้าหน้าที่ไอทีมีเวลาปรับตัวเข้ากับวิธีใหม่ ๆ ในการนำเสนอ BI นอกจากนี้ยังอนุญาตให้บุคคลดังกล่าวสร้างความเชี่ยวชาญใน Power BI ได้

มีการประกาศอย่างเป็นทางการเพื่อประกาศว่า Power BI เป็นเครื่องมือ BI ที่ได้รับการอนุมัติสำหรับองค์กร ในช่วงครึ่งหลังของปี 2018 ดังนั้นงานพัฒนา BI ใหม่ทั้งหมดควรเกิดขึ้นใน Power BI ความพร้อมใช้งานของ Power BI พรีเมียมเป็นตัวขับเคลื่อนหลักในการตัดสินใจนี้ ณ ขณะนี้ องค์กรไม่สนับสนุนให้ใช้แพลตฟอร์ม BI เดิม และเริ่มมีการวางแผนสำหรับการเปลี่ยนแปลง

เริ่มทำงานเพื่อโยกย้ายเนื้อหาที่มีอยู่จากแพลตฟอร์ม BI เดิมไปยัง Power BI ในช่วงปลายปี 2019 ผู้ใช้งานในช่วงแรก ๆ บางรายโยกย้ายเนื้อหาได้อย่างรวดเร็ว ซึ่งช่วยสร้างแรงผลักดันมากยิ่งขึ้นด้วย Power BI ไปทั่วองค์กร จากนั้นเจ้าของเนื้อหาและผู้เขียนถูกขอให้เริ่มการเตรียมการเพื่อโยกย้ายไปยัง Power BI อย่างสมบูรณ์ภายในสิ้นปี 2020 องค์กรยังคงเผชิญกับความท้าทายที่เกี่ยวข้องกับทักษะ เวลา และเงินทุน—แม้ว่าจะไม่มีความท้าทายใดที่เกี่ยวข้องกับแพลตฟอร์มเทคโนโลยีก็ตามแต่

> [!IMPORTANT]
> Power BI ประสบความสำเร็จและยึดมั่นในองค์กร ก่อนที่หน่วยธุรกิจจะถูกขอให้ดำเนินการโยกย้ายอย่างเป็นทางการออกจากแพลตฟอร์ม BI เดิม

### <a name="prepare-to-handle-varying-responses"></a>เตรียมรับมือกับการตอบสนองที่แตกต่างกัน

ในองค์กรแบบกระจายอำนาจขนาดใหญ่นี้ มีระดับการเปิดรับและความเต็มใจที่จะย้ายไปยัง Power BI ที่แตกต่างกัน นอกเหนือจากความกังวลเกี่ยวกับเวลาและงบประมาณแล้ว ยังมีพนักงานที่ลงทุนอย่างมากในการสร้างทักษะในแพลตฟอร์ม BI เดิม ดังนั้น การประกาศเกี่ยวกับการกำหนดมาตรฐานบน Power BI ไม่ได้รับการต้อนรับจากทุกคน เนื่องจากหน่วยธุรกิจแต่ละหน่วยมีงบประมาณของตนเอง หน่วยธุรกิจแต่ละหน่วยจึงอาจท้าทายการตัดสินใจได้ เช่น หน่วยธุรกิจนี้ เนื่องจากการตัดสินใจเกี่ยวกับเครื่องมือไอทีเกิดขึ้นจากส่วนกลาง ส่งผลให้เกิดความท้าทายบางประการสำหรับผู้สนับสนุนผู้บริหารและผู้นำ BI ที่ต้องรับมือ

> [!IMPORTANT]
> การสื่อสารกับทีมผู้นำทั่วทั้งหน่วยธุรกิจมีความสำคัญอย่างยิ่ง เพื่อให้แน่ใจว่าทุกคนมีความเข้าใจประโยชน์ขององค์กรระดับสูงของการกำหนดมาตรฐานบน Power BI การสื่อสารที่มีประสิทธิภาพกลายเป็นสิ่งสำคัญยิ่งขึ้นเมื่อการโยกย้ายข้อมูลถูกดำเนินไป และใกล้ถึงวันที่ปลดใช้งานแพลตฟอร์ม BI เดิม

### <a name="focus-on-the-bigger-picture"></a>โฟกัสไปยังรูปภาพขนาดใหญ่

บริษัทพบว่าในขณะที่รายงานการโยกย้ายข้อมูลบางฉบับสามารถจำลองแบบเดิมได้อย่างใกล้เคียง แต่ไม่ใช่ทุกรายงานที่สามารถจำลองแบบใน Power BI ได้อย่างน่าเชื่อถือ แม้ว่าจะคาดการณ์ไว้แล้ว—เนื่องจากแพลตฟอร์ม BI ทั้งหมดมีความแตกต่างกัน ทำให้เห็นได้ชัดว่าต้องมีแนวคิดในการออกแบบที่แตกต่างออกไป

มีการให้คำแนะนำแก่ผู้เขียนเนื้อหา: ให้มุ่งเน้นไปที่การสร้างรายงานที่เหมาะสมกับวัตถุประสงค์ใน Power BI แทนที่จะพยายามจำลองแบบที่แน่นอนของรายงานเดิม ด้วยเหตุนี้ ผู้เชี่ยวชาญเฉพาะด้านจะต้องพร้อมให้บริการอย่างแข็งขันในระหว่างขั้นตอนการโยกย้ายเพื่อขอคำปรึกษาและตรวจสอบความถูกต้อง มีการใช้ความพยายามในการพิจารณาวัตถุประสงค์ในการออกแบบรายงานและปรับปรุงตามความเหมาะสม

> [!IMPORTANT]
> บางครั้งแนวทางที่ดีกว่าคือการปรับปรุงระหว่างการโยกย้ายข้อมูล ในบางครั้ง ทางเลือกที่ดีกว่าคือการส่งมอบค่าที่แน่นอนเหมือนเดิม—โดยไม่มีการปรับปรุงทีมีนัยสำคัญ—เพื่อไม่ให้เกิดอันตรายกับไทม์ไลน์การโยกย้ายข้อมูล

### <a name="cautiously-assess-priorities"></a>ประเมินลำดับความสำคัญอย่างรอบคอบ

การวิเคราะห์ของแพลตฟอร์ม BI เดิมได้ดำเนินการเพื่อทำความเข้าใจการใช้งานอย่างถ่องแท้ แพลตฟอร์ม BI เดิม มีรายงานที่เผยแพร่หลายพันฉบับซึ่งประมาณครึ่งหนึ่งถูกเข้าถึงไปในปีที่แล้ว จำนวนดังกล่าวอาจถูกลดลงครึ่งหนึ่งอีกครั้งเมื่อมีการประเมินว่ารายงานใดที่ถือว่าส่งมอบค่าที่สำคัญให้กับองค์กร รายงานเหล่านี้ได้รับการจัดลำดับความสำคัญก่อนเป็นอันดับแรกสำหรับการโยกย้ายข้อมูล

> [!IMPORTANT]
> เป็นเรื่องง่ายมากที่จะประเมินค่าสูงเกินจริงว่ารายงานมีความสำคัญเพียงใด สำหรับรายงานที่ไม่ได้ถูกใช้บ่อย ให้ประเมินว่าสามารถปลดประจำการทั้งหมดได้หรือไม่ ในบางครั้ง สิ่งที่ถูกและง่ายที่สุดคือการไม่ทำอะไร

### <a name="cautiously-assess-complexity"></a>ประเมินความซับซ้อนอย่างระมัดระวัง

จากรายงานที่จัดลำดับความสำคัญเป็นอันดับแรกมีการรวบรวมการประมาณเวลาตามระดับความพยายามโดยประมาณ: ปกติ ปานกลาง หรือซับซ้อน แม้ว่าจะฟังดูเป็นกระบวนการที่ค่อนข้างตรงไปตรงมา แต่อย่าคาดหวังว่าการประมาณเวลาจะมีความแม่นยำในแต่ละรายงาน คุณอาจพบว่าการประมาณค่าอาจไม่ถูกต้องอย่างมาก ตัวอย่างเช่น บริษัทรายงานว่ามีความซับซ้อนสูง ซึ่งจะได้รับการสนทนาโดยที่ปรึกษาประมาณ 50 วัน อย่างไรก็ตาม รายงานที่ออกแบบใหม่ใน Power BI จะเสร็จสมบูรณ์ในเวลาประมาณ 50 ชั่วโมง

> [!IMPORTANT]
> แม้ว่าการประมาณเวลามักจำเป็นในการขอรับเงินทุนและการมอบหมายบุคลากร แต่ก็อาจมีค่ามากที่สุดในผลรวม

### <a name="decide-how-change-management-is-handled"></a>ตัดสินใจวิธีการจัดการการเปลี่ยนแปลง

ด้วยเนื้อหา BI จำนวนมากเช่นนี้ การจัดการการเปลี่ยนแปลงสำหรับรายงานที่เป็นเจ้าของธุรกิจจึงเป็นความท้าทาย รายงานที่จัดการด้วยไอทีจะได้รับการจัดการตามแนวทางการจัดการการเปลี่ยนแปลงมาตรฐาน อย่างไรก็ตาม เนื่องจากมีปริมาณมากจึงไม่สามารถผลักดันให้เกิดการเปลี่ยนแปลงจากศูนย์กลางสำหรับเนื้อหาที่เป็นเจ้าของธุรกิจได้

> [!IMPORTANT]
> ความรับผิดชอบเพิ่มเติมจะตกอยู่กับหน่วยธุรกิจ เมื่อไม่สามารถจัดการการเปลี่ยนแปลงจากทีมส่วนกลางเพียงทีมเดียวได้

### <a name="create-an-internal-community"></a>สร้างชุมชนภายใน

บริษัทได้จัดตั้งศูนย์ความเป็นเลิศ (COE) เพื่อจัดหาชั้นเรียนและทรัพยากรการฝึกอบรมภายใน COE ยังทำหน้าที่เป็นกลุ่มที่ปรึกษาภายในที่พร้อมให้ความช่วยเหลือผู้เขียนเนื้อหาเกี่ยวกับปัญหาทางเทคนิค ความละเอียดของอุปสรรค และแนวทางปฏิบัติที่ดีที่สุด

นอกจากนี้ยังมีชุมชน Power BI ภายใน ซึ่งประสบความสำเร็จอย่างมากเนื่องจากมีสมาชิกเกิน 1,600 คน ชุมชนได้รับการจัดการโดย Yammer สมาชิกสามารถถามคำถามที่เกี่ยวข้องเป็นการภายในและรับคำตอบตามแนวทางปฏิบัติที่ดีที่สุดและอยู่ในกรอบข้อจำกัดขององค์กร ประเภทการโต้ตอบระหว่างผู้ใช้กับผู้ใช้นี้ช่วยลดภาระการสนับสนุนจาก COE ได้มาก อย่างไรก็ตาม COE จะตรวจสอบคำถามและคำตอบและจะมีส่วนร่วมในการสนทนาตามความเหมาะสม

ส่วนขยายของชุมชนภายในคือเครือข่ายผู้เชี่ยวชาญ Power BI ที่ใหม่กว่า ซึ่งประกอบด้วยผู้ชนะ Power BI ที่เลือกไว้ล่วงหน้าจำนวนเล็กน้อยจากภายในองค์กร พวกเขาเป็นผู้ปฏิบัติงาน Power BI ที่มีทักษะสูงจากหน่วยธุรกิจ เป็นผู้ที่มีความกระตือรือร้นและต้องการแก้ปัญหาความท้าทายภายในธุรกิจ สมาชิกของเครือข่ายผู้เชี่ยวชาญ Power BI คาดว่าจะปฏิบัติตามแนวทางปฏิบัติที่ดีที่สุดและแนวทางที่กำหนดโดย COE และช่วยให้ชุมชน Power BI ภายในวงกว้างเข้าใจและนำไปใช้ได้ แม้ว่าเครือข่ายผู้เชี่ยวชาญของ Power BI จะทำงานร่วมกับ COE และสามารถรับการฝึกอบรมโดยเฉพาะก็ตาม ผู้เชี่ยวชาญของ Power BI จะทำการดำเนินการโดยอิสระจาก COE ผู้เชี่ยวชาญ Power BI แต่ละคนอาจกำหนดพารามิเตอร์สำหรับวิธีการทำงาน โดยคำนึงถึงความรับผิดชอบและลำดับความสำคัญอื่น ๆ ในบทบาทอย่างเป็นทางการของตน

> [!IMPORTANT]
> มีการขอบเขตที่กำหนดไว้เป็นอย่างดีสำหรับสิ่งที่ COE ทำ เช่น: การเริ่มนำไปใช้ นโยบายการกำกับดูแล การแนะนำ แนวทางปฏิบัติ การฝึกอบรม การสนับสนุน และอาจถึงขั้นลงมือพัฒนา ในขณะที่ COE มีค่าอย่างเหลือเชื่อ การวัดผลตอบแทนจากการลงทุนอาจเป็นเรื่องที่ยาก

### <a name="monitor-migration-progress-and-success"></a>ติดตามความคืบหน้าและความสำเร็จในการโยกย้ายข้อมูล

ดัชนีชี้วัดผลการปฏิบัติงานหลัก (KPI) จะมีการถูกตรวจสอบอย่างต่อเนื่องในระหว่างการโยกย้ายไปยัง Power BI ช่วยให้บริษัทเข้าใจแนวโน้มของเมตริก เช่น จำนวนการเข้าชมรายงาน จำนวนรายงานที่ใช้งานอยู่ และผู้ใช้ที่แตกต่างกันต่อเดือน การใช้งาน Power BI ที่เพิ่มขึ้นจะวัดควบคู่ไปกับการใช้งานแพลตฟอร์ม BI เดิมที่ลดลงโดยมีเป้าหมายในการบรรลุความสัมพันธ์แบบผกผัน มีการอัปเดตเป้าหมายในแต่ละเดือนเพื่อปรับให้เข้ากับการเปลี่ยนแปลง หากการใช้งานไม่เกิดขึ้นตามจังหวะที่ต้องการจะมีการระบุปัญหาคอขวดเพื่อให้สามารถดำเนินการได้อย่างเหมาะสม

> [!IMPORTANT]
> สร้างดัชนีชี้วัดการโยกย้ายข้อมูลด้วยระบบธุรกิจอัจฉริยะที่สามารถดำเนินการได้เพื่อตรวจสอบความสำเร็จของความพยายามในการโยกย้ายข้อมูล

## <a name="large-transportation-and-logistics-company"></a>บริษัทขนส่งและโลจิสติกส์ขนาดใหญ่

บริษัทขนส่งและโลจิสติกส์รายใหญ่ในอเมริกาเหนือกำลังลงทุนอย่างจริงจังในการปรับปรุงโครงสร้างพื้นฐานข้อมูลและระบบวิเคราะห์ให้ทันสมัย

### <a name="allow-a-period-of-gradual-growth"></a>อนุมัติช่วงเวลาของการเติบโตอย่างค่อยเป็นค่อยไป

บริษัทได้เริ่มใช้ Power BI ในปี 2018 ในช่วงกลางปี 2019 Power BI กลายเป็นแพลตฟอร์มที่ต้องการสำหรับกรณีการใช้งาน BI ใหม่ทั้งหมด จากนั้นในปี 2020 บริษัทมุ่งเน้นไปที่แพลตฟอร์ม BI ที่มีอยู่นอกเหนือไปจากโซลูชัน ASP.NET BI ที่พัฒนาขึ้นเองที่หลากหลาย

> [!IMPORTANT]
> Power BI มีผู้ใช้ที่ใช้งานอยู่จำนวนมากทั่วทั้งองค์กร ก่อนที่จะเริ่มขั้นตอนการออกจากแพลตฟอร์มและโซลูชัน BI เดิม

### <a name="balance-centralized-and-distributed-groups"></a>สร้างสมดุลรวมศูนย์และกลุ่มแบบกระจาย

ในบริษัท จะมีทีม BI อยู่สองประเภท: ทีม BI ส่วนกลางและกลุ่มการวิเคราะห์ที่กระจายอยู่ทั่วทั้งองค์กร ทีม BI ส่วนกลางมีความรับผิดชอบในการเป็นเจ้าของสำหรับ Power BI ในฐานะแพลตฟอร์ม แต่ไม่ได้เป็นเจ้าของเนื้อหาใด ๆ ด้วยวิธีนี้ ทีม BI ส่วนกลางเป็นฮับการเปิดใช้งานทางเทคนิคที่สนับสนุนกลุ่มการวิเคราะห์แบบกระจาย

กลุ่มการวิเคราะห์แต่ละกลุ่มมีไว้สำหรับหน่วยธุรกิจเฉพาะหรือฟังก์ชันบริการที่ใช้ร่วมกัน กลุ่มเล็ก ๆ อาจมีนักวิเคราะห์คนเดียวในขณะที่กลุ่มใหญ่สามารถมีนักวิเคราะห์ 10-15 คนได้

> [!IMPORTANT]
> กลุ่มการวิเคราะห์แบบกระจายประกอบด้วยผู้เชี่ยวชาญเฉพาะเรื่องที่คุ้นเคยกับความต้องการทางธุรกิจในแต่ละวัน การแยกนี้ช่วยให้ทีม BI ส่วนกลางมุ่งเน้นไปที่การเปิดใช้งานทางเทคนิคและการสนับสนุนของบริการและเครื่องมือ BI เป็นหลัก

### <a name="focus-on-dataset-reusability"></a>มุ่งเน้นไปที่การใช้ซ้ำของชุดข้อมูล

การพึ่งพาโซลูชัน ASP.NET BI ที่กำหนดเองเป็นอุปสรรคในการพัฒนาโซลูชัน BI ใหม่ ชุดทักษะที่จำเป็นหมายความว่าผู้เขียนเนื้อหาแบบบริการตนเองมีจำนวนที่น้อย เนื่องจาก Power BI เป็นเครื่องมือที่เข้าถึงได้ง่ายกว่ามาก—ออกแบบมาโดยเฉพาะสำหรับ BI แบบบริการตนเอง—จะแพร่กระจายไปทั่วทั้งองค์กรอย่างรวดเร็วทันทีที่เปิดตัว

การเพิ่มขีดความสามารถของนักวิเคราะห์ข้อมูลภายในบริษัททำให้เกิดผลลัพธ์ที่เป็นบวกทันที อย่างไรก็ตาม การมุ่งเน้นที่การพัฒนา Power BI ระดับเริ่มต้นอยู่ที่การแสดงภาพ แม้ว่าจะส่งผลให้โซลูชัน BI มีค่า แต่การมุ่งเน้นนี้ก็ส่งผลต่อไฟล์ Power BI Desktop จำนวนมาก ซึ่งแต่ละไฟล์มีความสัมพันธ์แบบหนึ่งต่อหนึ่งระหว่างรายงานและชุดข้อมูล ส่งผลให้มีชุดข้อมูลจำนวนมากและการทำซ้ำข้อมูลและตรรกะทางธุรกิจ เพื่อลดความซ้ำซ้อนของข้อมูล ตรรกะ และความพยายาม  บริษัทได้จัดส่งการฝึกอบรมและให้การสนับสนุนแก่ผู้เขียนเนื้อหา

> [!IMPORTANT]
> รวมถึงข้อมูลเกี่ยวกับความสำคัญของการนำข้อมูลกลับมาใช้ใหม่ในการฝึกอบรมภายในของคุณ กล่าวถึงแนวคิดที่สำคัญโดยเร็วที่สุดเท่าที่จะทำได้จริง

### <a name="test-data-access-multiple-ways"></a>ทดสอบการเข้าถึงข้อมูลแบบหลายวิธี

แพลตฟอร์มคลังข้อมูลของบริษัทคือ DB2 อิงจากการออกแบบคลังข้อมูลปัจจุบัน บริษัทได้พบแบบจำลอง DirectQuery—แทนที่จะเป็นแบบจำลองการนำเข้า—ที่ทำงานได้ดีที่สุดสำหรับความต้องการ

> [!IMPORTANT]
> ดำเนินการพิสูจน์แนวคิดทางเทคนิคเพื่อประเมินโหมดการจัดเก็บแบบจำลองที่ทำงานได้ดีที่สุด อีกทั้งยังสอนผู้สร้างโมเดลข้อมูลเกี่ยวกับโหมดการจัดเก็บโมเดลและวิธีเลือกโหมดที่เหมาะสมสำหรับโปรเจ็กต์อีกด้วย

### <a name="educate-authors-about-premium-licensing"></a>ให้ความรู้แก่ผู้เขียนเกี่ยวกับสิทธิ์การใช้งานระดับพรีเมียม

เนื่องจากการเริ่มต้นใช้งาน Power BI ทำได้ง่ายขึ้น (เทียบกับแพลตฟอร์ม BI เดิม) ผู้ใช้งานในช่วงแรกหลายคนเป็นผู้ที่ไม่มีสิทธิการใช้งานสำหรับเครื่องมือ BI ก่อนหน้านี้ ตามที่คาดไว้ จำนวนผู้เขียนเนื้อหาเพิ่มขึ้นอย่างมาก ผู้เขียนเนื้อหาเหล่านี้ต้องการแบ่งปันเนื้อหากับผู้อื่นอย่างเข้าใจซึ่งส่งผลให้เกิดความต้องการสิทธิ์การใช้งาน Power BI Pro เพิ่มเติมอย่างต่อเนื่อง

บริษัทได้ลงทุนสร้างพื้นที่ทำงานระดับพรีเมียมขนาดใหญ่ โดยเฉพาะอย่างยิ่งในการแจกจ่ายเนื้อหา Power BI ให้กับผู้ใช้จำนวนมากด้วยสิทธิ์การใช้งานฟรีของ Power BI ทีมสนับสนุนจะทำงานร่วมกับผู้เขียนเนื้อหาเพื่อให้แน่ใจว่าพวกเขาใช้พื้นที่ทำงานระดับพรีเมียมตามความเหมาะสม เพื่อหลีกเลี่ยงการจัดสรรสิทธิ์การใช้งาน Power BI Pro โดยไม่จำเป็นเมื่อผู้ใช้ต้องการใช้เนื้อหาเท่านั้น

> [!IMPORTANT]
> คำถามที่พบบ่อยในการให้สิทธิ์การใช้งาน เตรียมพร้อมที่จะให้ความรู้และช่วยเหลือผู้เขียนเนื้อหาในการตอบคำถามเกี่ยวกับสิทธิ์การใช้งาน ตรวจสอบว่าคำขอของผู้ใช้สำหรับสิทธิ์การใช้งาน Power BI Pro นั้นมีความเป็นธรรม

### <a name="understand-the-data-gateways"></a>ทำความเข้าใจกับเกตเวย์ข้อมูล

ในช่วงต้น บริษัทมีเกตเวย์ส่วนบุคคลอยู่มากมาย การใช้คลัสเตอร์เกตเวย์ข้อมูลภายในองค์กรจะเปลี่ยนความพยายามในการจัดการไปยังทีม BI ส่วนกลาง ซึ่งช่วยให้ชุมชนผู้เขียนเนื้อหามุ่งเน้นไปที่การผลิตเนื้อหา ทีม BI ส่วนกลางทำงานร่วมกับชุมชนผู้ใช้ Power BI ภายในเพื่อลดจำนวนเกตเวย์ส่วนบุคคล

> [!IMPORTANT]
> มีแผนสำหรับการสร้างและจัดการเกตเวย์ข้อมูลภายในองค์กร ตัดสินใจว่าใครที่ได้รับอนุญาตให้ติดตั้งและใช้เกตเวย์ส่วนบุคคลและบังคับใช้กับนโยบายเกตเวย์

### <a name="formalize-your-support-plan"></a>จัดทำแผนสนับสนุนของคุณอย่างเป็นทางการ

เนื่องจากการใช้ Power BI เพิ่มขึ้นภายในองค์กรบริษัท พบว่ามีแนวทางการสนับสนุนหลายชั้นทำงานได้ดี:

- **ระดับชั้น 1: ภายในทีม:** ผู้คนเรียนรู้จากและสอนซึ่งกันและกันในแต่ละวัน
- **ระดับชั้น 2: ชุมชน Power BI:** ผู้คนถามคำถามของทีมชุมชนภายในเพื่อเรียนรู้จากกันและกันและมีการสื่อสารข้อมูลสำคัญ
- **ระดับชั้น 3: ทีมศูนย์กลาง BI และ COE:** ผู้คนส่งคำขอทางอีเมลเพื่อขอความช่วยเหลือ _เวลาทำการ_ การประชุมจะจัดขึ้นสองครั้งต่อสัปดาห์เพื่อหารือเกี่ยวกับปัญหาและแบ่งปันความคิดร่วมกัน

> [!IMPORTANT]
> แม้ว่าระดับชั้นสองชั้นแรกจะดูเป็นทางการน้อยกว่า แต่ก็มีความสำคัญเท่าเทียมกับระดับชั้นที่สามของการสนับสนุน ผู้ใช้ที่มีประสบการณ์มักจะพึ่งพาคนที่พวกเขารู้จักเป็นส่วนใหญ่ในขณะที่ผู้ใช้ใหม่ ๆ (หรือผู้ที่เป็นนักวิเคราะห์ข้อมูลรายเดียวสำหรับหน่วยธุรกิจหรือบริการที่ใช้ร่วมกัน) มีแนวโน้มที่จะพึ่งพาการสนับสนุนอย่างเป็นทางการมากกว่า

### <a name="invest-in-training-and-governance"></a>การลงทุนในการฝึกอบรมและการกำกับดูแล

ในปีที่ผ่านมา บริษัทได้ปรับปรุงข้อเสนอการฝึกอบรมภายในและปรับปรุงโปรแกรมการกำกับดูแลข้อมูล คณะกรรมการกำกับดูแลประกอบด้วยสมาชิกหลักจากกลุ่มการวิเคราะห์แบบกระจายแต่ละกลุ่มรวมทั้ง COE

ขณะนี้มีหลักสูตร Power BI ภายในหกหลักสูตรในแค็ตตาล็อกภายใน หลักสูตร[แดชบอร์ดในหนึ่งวัน](https://powerbi.microsoft.com/diad/)ยังคงเป็นหลักสูตรยอดนิยมสำหรับผู้เริ่มต้น เพื่อช่วยให้ผู้ใช้พัฒนาทักษะได้ลึกขึ้น ได้มีการนำเสนอชุดของหลักสูตร Power BI สามหลักสูตรและหลักสูตร DAX สองหลักสูตร

หนึ่งในการตัดสินใจด้านนโยบายการกำกับดูแลข้อมูลที่สำคัญที่สุดที่เกี่ยวข้องกับการจัดการความสามารถระดับพรีเมียม บริษัทเลือกที่จะปรับความจุของตนให้สอดคล้องกับพื้นที่การวิเคราะห์ที่สำคัญในหน่วยธุรกิจและบริการที่ใช้ร่วมกัน ดังนั้น ถ้าหากมีความไร้ประสิทธิภาพผลกระทบจะเกิดขึ้นภายในพื้นที่เดียวเท่านั้นและผู้ดูแลระบบความสามารถแบบกระจายอำนาจจะได้รับอำนาจในการจัดการความสามารถตามที่เห็นสมควร

> [!IMPORTANT]
> ใส่ใจกับวิธีใช้ความจุระดับพรีเมียมและวิธีกำหนดพื้นที่ทำงานให้กับพวกเขา
