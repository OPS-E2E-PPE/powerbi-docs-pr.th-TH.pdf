# <a name="azure-security-baseline-for-power-bi"></a>ข้อมูลพื้นฐานด้านความปลอดภัย Azure สำหรับ Power BI

ข้อมูลพื้นฐานด้านความปลอดภัยนี้ปรับใช้คำแนะนำจาก [Azure Security Benchmark เวอร์ชัน 2.0](/azure/security/benchmarks/overview) กับ Power BI Azure Security Benchmark ให้คำแนะนำเกี่ยวกับวิธีการที่คุณสามารถรักษาความปลอดภัยโซลูชันระบบคลาวด์ของคุณบน Azure มีการจัดกลุ่มเนื้อหาตาม **ตัวควบคุมความปลอดภัย** ที่กำหนดโดย Azure Security Benchmark และคำแนะนำที่เกี่ยวข้อง ซึ่งใช้ได้กับ Power BI ไม่รวม **ตัวควบคุม** ที่ใช้ไม่ได้กับ Power BI

หากต้องการดูว่า Power BI แมปกับ Azure Security Benchmark โดยสมบูรณ์ได้อย่างไร โปรดดูที่[ไฟล์การแมปข้อมูลพื้นฐานด้านความปลอดภัย Power BI ฉบับเต็ม](https://github.com/MicrosoftDocs/SecurityBenchmarks/tree/master/Azure%20Offer%20Security%20Baselines)

**คำแนะนำ**: Power BI สนับสนุนการเชื่อมต่อผู้เช่า Power BI ของคุณกับจุดสิ้นสุดลิงก์ส่วนตัว และปิดใช้งานการเข้าถึงอินเทอร์เน็ตสาธารณะ

- [ใช้ลิงก์ส่วนตัวสำหรับการเข้าใช้งาน Power BI](../admin/service-security-private-links.md)

## <a name="identity-management"></a>การจัดการข้อมูลเอกลักษณ์

*สำหรับข้อมูลเพิ่มเติม โปรดดูที่ [Azure Security Benchmark: การจัดการข้อมูลเอกลักษณ์](/azure/security/benchmarks/security-controls-v2-identity-management)*

### <a name="im-1-standardize-azure-active-directory-as-the-central-identity-and-authentication-system"></a>IM-1: ทำให้ Azure Active Directory เป็นมาตรฐานในฐานะข้อมูลเอกลักษณ์ส่วนกลางและระบบการรับรองความถูกต้อง

**คำแนะนำ**: Power BI ถูกรวมเข้ากับ Azure Active Directory (Azure AD) ซึ่งเป็นข้อมูลเอกลักษณ์เริ่มต้นและบริการจัดการการเข้าถึงของ Azure คุณควรสร้างมาตรฐานบน Azure AD เพื่อควบคุมข้อมูลเอกลักษณ์ขององค์กรและการจัดการการเข้าถึง

การรักษาความปลอดภัย Azure AD ควรมีลำดับความสำคัญสูงในแนวทางปฏิบัติด้านความปลอดภัยบนคลาวด์ขององค์กรของคุณ Azure AD ให้คะแนนความปลอดภัยของข้อมูลเอกลักษณ์เพื่อช่วยคุณประเมินสภาวะความปลอดภัยของข้อมูลเอกลักษณ์ที่สัมพันธ์กับคำแนะนำแนวทางปฏิบัติที่ดีที่สุดของ Microsoft ใช้คะแนนเพื่อวัดว่าการกำหนดค่าของคุณตรงกับคำแนะนำแนวทางปฏิบัติที่ดีที่สุดเพียงใดและเพื่อปรับปรุงสภาวะความปลอดภัยของคุณ

หมายเหตุ: Azure AD สนับสนุนข้อมูลประจำตัวภายนอกที่อนุญาตให้ผู้ใช้ที่ไม่มีบัญชี Microsoft สามารถลงชื่อเข้าใช้แอปพลิเคชันและทรัพยากรของพวกเขาด้วยข้อมูลประจำตัวภายนอกได้

- [การเช่าใน Azure Active Directory](/azure/active-directory/develop/single-and-multi-tenant-apps)

- [วิธีการสร้างและกำหนดค่าอินสแตนซ์ Azure AD](/azure/active-directory/fundamentals/active-directory-access-create-new-tenant)

- [ใช้ผู้ให้บริการข้อมูลเอกลักษณ์ภายนอกสำหรับแอปพลิเคชัน](/azure/active-directory/b2b/identity-providers)

- [คะแนนความปลอดภัยของข้อมูลเอกลักษณ์ใน Azure Active Directory คืออะไร](/azure/active-directory/fundamentals/identity-secure-score)

**การตรวจสอบศูนย์รักษาความปลอดภัย Azure**: ไม่สามารถใช้งานได้

**ความรับผิดชอบ**: ลูกค้า

### <a name="im-2-manage-application-identities-securely-and-automatically"></a>IM-2: จัดการข้อมูลเอกลักษณ์ของแอปพลิเคชันอย่างปลอดภัยและโดยอัตโนมัติ

**คำแนะนำ**: Power BI และ Power BI Embedded สนับสนุนการใช้องค์ประกอบหลักของบริการ จัดเก็บข้อมูลประจำตัวขององค์ประกอบหลักของบริการที่ใช้สำหรับเข้ารหัสหรือเข้าถึง Power BI ใน Key Vault กำหนดนโยบายการเข้าถึงที่เหมาะสมให้กับห้องนิรภัย (Vault) และตรวจสอบสิทธิ์การเข้าถึงอย่างสม่ำเสมอ

ทำให้พื้นที่ทำงานและชุดข้อมูลระดับ Premium ทำงานโดยอัตโนมัติด้วยองค์ประกอบหลักของบริการ https://docs.microsoft.com/power-bi/admin/service-premium-service-principal

**การตรวจสอบศูนย์รักษาความปลอดภัย Azure**: ไม่สามารถใช้งานได้

**ความรับผิดชอบ**: ลูกค้า

### <a name="im-3-use-azure-ad-single-sign-on-sso-for-application-access"></a>IM-3: ใช้ลงชื่อเข้าระบบครั้งเดียว (single sign-on : SSO) ของ Azure AD สำหรับการเข้าถึงแอปพลิเคชัน

**คำแนะนำ**: Power BI ใช้ Azure Active Directory เพื่อจัดเตรียมข้อมูลเอกลักษณ์และการจัดการการเข้าถึงทรัพยากร Azure แอปพลิเคชันระบบคลาวด์ และแอปพลิเคชันภายในองค์กร ซึ่งรวมถึงข้อมูลเอกลักษณ์ขององค์กร เช่น พนักงาน ตลอดจนข้อมูลเอกลักษณ์ภายนอก เช่น คู่ค้า ผู้ขาย และซัพพลายเออร์ ซึ่งเป็นการเปิดใช้งานการลงชื่อเข้าระบบครั้งเดียว (SSO) เพื่อจัดการและรักษาความปลอดภัยการเข้าถึงข้อมูลและทรัพยากรขององค์กรของคุณภายในองค์กรและในระบบคลาวด์ เชื่อมต่อผู้ใช้ แอปพลิเคชัน และอุปกรณ์ทั้งหมดของคุณกับ Azure AD เพื่อการเข้าถึงที่ราบรื่น ปลอดภัย และความสามารถในการมองเห็นและการควบคุมที่ดีขึ้น

- [ทำความเข้าใจ SSO สำหรับแอปพลิเคชันด้วย Azure AD](/azure/active-directory/manage-apps/what-is-single-sign-on)

**การตรวจสอบศูนย์รักษาความปลอดภัย Azure**: ไม่สามารถใช้งานได้

**ความรับผิดชอบ**: ลูกค้า

### <a name="im-4-use-strong-authentication-controls-for-all-azure-active-directory-based-access"></a>IM-4: ใช้ตัวควบคุมการรับรองความถูกต้องที่เข้มงวดสำหรับการเข้าถึงที่ใช้ Azure Active Directory ทั้งหมด

**คำแนะนำ**: Power BI ถูกรวมเข้ากับ Azure AD ที่สนับสนุนตัวควบคุมการรับรองความถูกต้องที่เข้มงวดผ่านการรับรองความถูกต้องโดยใช้หลายปัจจัย (MFA) และวิธีการแบบไม่ใช้รหัสผ่านที่เข้มงวด
- การรับรองความถูกต้องโดยใช้หลายปัจจัย - เปิดใช้งาน Azure AD MFA และปฏิบัติตามคำแนะนำ Azure Security Center Identity และ Access Management สำหรับแนวทางปฏิบัติที่ดีที่สุดบางประการในการตั้งค่า MFA ของคุณ คุณสามารถบังคับใช้ MFA ได้กับทุกคน เลือกผู้ใช้หรือในระดับต่อผู้ใช้ตามเงื่อนไขการลงชื่อเข้าใช้และปัจจัยความเสี่ยง
- การรับรองความถูกต้องแบบไม่ใช้รหัสผ่าน - ตัวเลือกการรับรองความถูกต้องแบบไม่ใช้รหัสผ่านมีให้เลือกสามแบบ Windows Hello for Business, แอป Microsoft Authenticator และวิธีวิธีการรับรองความถูกต้องในองค์กร เช่น สมาร์ทการ์ด

สำหรับผู้ดูแลระบบและผู้ใช้ที่มีสิทธิ์พิเศษ ตรวจสอบให้แน่ใจว่ามีการใช้การรับรองความถูกต้องที่เข้มงวดระดับสูงสุด ตามด้วยการนำนโยบายการรับรองความถูกต้องที่เข้มงวดและเหมาะสมไปใช้กับผู้ใช้รายอื่น

หมายเหตุ: คุณสามารถบังคับใช้ MFA สำหรับบัญชีผู้ใช้ที่เปิดใช้งานใน Azure AD เท่านั้น องค์ประกอบหลักของบริการของ Power BI ไม่สนับสนุนการใช้งาน MFA

- [วิธีเปิดใช้งาน MFA ใน Azure](/azure/active-directory/authentication/howto-mfa-getstarted)

- [บทนำเกี่ยวกับตัวเลือกการรับรองความถูกต้องโดยไม่ใช้รหัสผ่านสำหรับ Azure Active Directory](/azure/active-directory/authentication/concept-authentication-passwordless)

**การตรวจสอบศูนย์รักษาความปลอดภัย Azure**: ไม่สามารถใช้งานได้

**ความรับผิดชอบ**: ลูกค้า

### <a name="im-5-monitor-and-alert-on-account-anomalies"></a>IM-5: ตรวจสอบและแจ้งเตือนเกี่ยวกับความผิดปกติของบัญชี

**คำแนะนำ**: กำหนดนโยบายการตรวจจับความผิดปกติใน Microsoft Cloud App Security ที่สามารถกำหนดขอบเขตได้อย่างอิสระ เพื่อที่จะปรับใช้เฉพาะกับผู้ใช้และกลุ่มที่คุณต้องการรวม นโยบายการตรวจจับความผิดปกติเหล่านี้สามารถช่วยตรวจหาและตรวจสอบความผิดปกติของพฤติกรรมที่เกี่ยวข้องกับผู้ใช้ที่เข้าถึงและใช้ Power BI

- [ใช้ตัวควบคุม Microsoft Cloud App Security ใน Power BI](../admin/service-security-using-microsoft-cloud-app-security-controls.md)

**การตรวจสอบศูนย์รักษาความปลอดภัย Azure**: ไม่สามารถใช้งานได้

**ความรับผิดชอบ**: ลูกค้า

### <a name="im-6-restrict-azure-resource-access-based-on-conditions"></a>IM-6: จำกัดการเข้าถึงทรัพยากร Azure โดยยึดตามเงื่อนไข

**คำแนะนำ**: Power BI สนับสนุนการเข้าถึงแบบมีเงื่อนไขของ Azure AD สำหรับการควบคุมการเข้าถึงที่ละเอียดยิ่งขึ้นตามเงื่อนไขที่ผู้ใช้กำหนด เช่น การเข้าสู่ระบบของผู้ใช้จากช่วง IP บางช่วงจะต้องใช้ MFA สำหรับการเข้าสู่ระบบ นโยบายการจัดการเซสชันการรับรองความถูกต้องแบบละเอียดยังสามารถใช้สำหรับกรณีการใช้งานที่แตกต่างกันอีกด้วย

- [ภาพรวมการเข้าถึงแบบมีเงื่อนไขของ Azure](/azure/active-directory/conditional-access/overview)

- [นโยบายการเข้าถึงแบบมีเงื่อนไขทั่วไป](/azure/active-directory/conditional-access/concept-conditional-access-policy-common)

- [กำหนดค่าการจัดการเซสชันการรับรองความถูกต้องด้วยการเข้าถึงแบบมีเงื่อนไข](/azure/active-directory/conditional-access/howto-conditional-access-session-lifetime)

- [ใช้ตัวควบคุม Microsoft Cloud App Security ใน Power BI](../admin/service-security-using-microsoft-cloud-app-security-controls.md)

**การตรวจสอบศูนย์รักษาความปลอดภัย Azure**: ไม่สามารถใช้งานได้

### <a name="im-7-eliminate-unintended-credential-exposure"></a>IM-7: กำจัดการเปิดเผยข้อมูลประจำตัวโดยไม่ได้ตั้งใจ

**คำแนะนำ**: สำหรับแอปพลิเคชันแบบฝังตัวของ Power BI ขอแนะนำให้ติดตั้งเครื่องสแกนข้อมูลประจำตัว (Credential Scanner) เพื่อระบุข้อมูลประจำตัวภายในโค้ดของคุณ นอกจากนี้ เครื่องสแกนข้อมูลประจำตัวจะสนับสนุนให้ย้ายข้อมูลประจำตัวที่ค้นพบไปยังตำแหน่งที่ปลอดภัยมากขึ้น เช่น Azure Key Vault

จัดเก็บคีย์การเข้ารหัสหรือข้อมูลประจำตัวขององค์ประกอบหลักของบริการที่ใช้สำหรับเข้ารหัสหรือเข้าถึง Power BI ใน Key Vault กำหนดนโยบายการเข้าถึงที่เหมาะสมให้กับห้องนิรภัย (Vault) และตรวจสอบสิทธิ์การเข้าถึงอย่างสม่ำเสมอ
 
สำหรับ GitHub คุณสามารถใช้คุณลักษณะการสแกนข้อมูลลับแบบเนทีฟเพื่อระบุข้อมูลประจำตัวหรือข้อมูลลับรูปแบบอื่น ๆ ภายในโค้ด

- [นำคีย์การเข้ารหัสลับของคุณเองมาใช้กับ Power BI](../admin/service-encryption-byok.md)

## <a name="privileged-access"></a>การเข้าถึงที่มีสิทธิ์พิเศษ

*สำหรับข้อมูลเพิ่มเติม โปรดดูที่ [Azure Security Benchmark: การเข้าถึงที่มีสิทธิ์พิเศษ](/azure/security/benchmarks/security-controls-v2-privileged-access)*

### <a name="pa-1-protect-and-limit-highly-privileged-users"></a>PA-1: ป้องกันและจำกัดผู้ใช้ที่มีสิทธิ์พิเศษสูง

**คำแนะนำ**: เพื่อลดความเสี่ยงและปฏิบัติตามหลักการของสิทธิ์พิเศษน้อยที่สุด เราขอแนะนำให้คงความเป็นสมาชิกของผู้ดูแลระบบ Power BI ไว้กับคนจำนวนน้อย ผู้ใช้ที่มีสิทธิ์พิเศษเหล่านี้อาจสามารถเข้าถึงและปรับเปลี่ยนคุณลักษณะการจัดการทั้งหมดสำหรับองค์กรได้ ผู้ดูแลระบบส่วนกลางมีสิทธิ์ของผู้ดูแลระบบในบริการ Power BI โดยปริยายเช่นกันผ่าน Microsoft 365 หรือ Azure Active Directory

Power BI มีบัญชีผู้ใช้ที่มีสิทธิ์พิเศษสูงด้านล่าง:
- ผู้ดูแลระบบส่วนกลาง
- ผู้ดูแลการเรียกเก็บเงิน
- ผู้ดูแลสิทธิการใช้งาน
- ผู้ดูแลระบบผู้ใช้
- ผู้ดูแลระบบ Power BI
- ผู้ดูแลระบบ Power BI Premium Capacity
- ผู้ดูแลระบบ Power BI Embedded Capacity

Power BI สนับสนุนนโยบายเซสชันใน Azure AD เพื่อเปิดใช้งานนโยบายการเข้าถึงแบบมีเงื่อนไขและเซสชันเส้นทางที่ใช้ใน Power BI ผ่านทางบริการของ Cloud App Security

- [บทบาทผู้ดูแลระบบที่เกี่ยวข้องกับ Power BI](../admin/service-admin-administering-power-bi-in-your-organization.md#administrator-roles-related-to-power-bi)

- [การจัดการการเข้าถึงที่มีสิทธิ์พิเศษ M365](/microsoft-365/compliance/privileged-access-management-overview?view=o365-worldwide&preserve-view=true)

**คำแนะนำ**: ในฐานะผู้ดูแลระบบบริการ Power BI คุณสามารถวิเคราะห์การใช้งานสำหรับทรัพยากร Power BI ทั้งหมดที่ระดับผู้เช่าโดยใช้รายงานแบบกำหนดเองโดยยึดตามบันทึกกิจกรรมของ Power BI คุณสามารถดาวน์โหลดกิจกรรมโดยใช้ REST API หรือ PowerShell cmdlet คุณยังสามารถกรองข้อมูลกิจกรรมตามช่วงวันที่ ผู้ใช้และประเภทกิจกรรมได้

**คำแนะนำ**: Power BI ถูกรวมเข้ากับ Azure Active Directory และ M365 เพื่อจัดการทรัพยากร เพื่อป้องกันการถูกล็อกจากผู้เช่า M365 หรือองค์กร Azure AD โดยไม่ได้ตั้งใจ ให้ตั้งค่าบัญชีการเข้าถึงฉุกเฉินสำหรับการเข้าถึงเมื่อไม่สามารถใช้บัญชีผู้ดูแลระบบปกติได้ บัญชีการเข้าถึงฉุกเฉินมักมีสิทธิ์พิเศษสูง และไม่ควรกำหนดให้กับบุคคลใดบุคคลหนึ่ง บัญชีการเข้าถึงฉุกเฉินถูกจำกัดไว้ในสถานการณ์ฉุกเฉินหรือ "กระจกแตก" ซึ่งไม่สามารถใช้บัญชีดูแลระบบปกติได้

### <a name="ir-2-preparation--setup-incident-notification"></a>IR-2: การเตรียมการ - ตั้งค่าการแจ้งเตือนเหตุการณ์

### <a name="ir-3-detection-and-analysis--create-incidents-based-on-high-quality-alerts"></a>IR-3: การตรวจจับและวิเคราะห์ - สร้างเหตุการณ์ตามการแจ้งเตือนคุณภาพสูง

**คำแนะนำ**: ตรวจสอบว่าคุณมีกระบวนการสร้างการแจ้งเตือนคุณภาพสูง และวัดคุณภาพของการแจ้งเตือน ซึ่งช่วยให้คุณสามารถเรียนรู้บทเรียนจากเหตุการณ์ที่ผ่านมา และจัดลำดับความสำคัญการแจ้งเตือนสำหรับนักวิเคราะห์เพื่อไม่ให้พวกเขาต้องเสียเวลากับผลบวกที่ผิดพลาด

ตรวจสอบการแจ้งเตือนที่เกี่ยวข้องกับ Power BI ใน Microsoft Cloud App Security การแจ้งเตือนคุณภาพสูงอาจเกิดขึ้นตามประสบการณ์จากเหตุการณ์ในอดีต แหล่งที่มาของชุมชนที่ผ่านการตรวจสอบแล้ว และเครื่องมือที่ออกแบบมาเพื่อสร้างและล้างการแจ้งเตือนโดยการหลอมรวมและเชื่อมโยงแหล่งสัญญาณที่หลากหลาย

- [ตรวจสอบการแจ้งเตือนใน Cloud App Security](/cloud-app-security/monitor-alerts)

### <a name="ir-4-detection-and-analysis--investigate-an-incident"></a>IR-4: การตรวจจับและวิเคราะห์ - ตรวจสอบเหตุการณ์

**คำแนะนำ**: สร้างคู่มือการตอบสนองต่อเหตุการณ์สำหรับองค์กรของคุณ ทำการฝึกซ้อมเพื่อทดสอบความสามารถในการตอบสนองต่อเหตุการณ์ของระบบของคุณด้วยการใช้งานปกติ ระบุจุดอ่อนและช่องว่างและแก้ไขแผนตามความจำเป็น

ตรวจสอบให้แน่ใจว่ามีแผนการตอบสนองต่อเหตุการณ์ที่เป็นลายลักษณ์อักษรที่กำหนดบทบาทของบุคลากรทั้งหมด ตลอดจนขั้นตอนของการรับมือ / การจัดการเหตุการณ์ตั้งแต่การตรวจจับไปจนถึงการทบทวนหลังเกิดเหตุ

### <a name="ir-5-detection-and-analysis--prioritize-incidents"></a>IR-5: การตรวจจับและวิเคราะห์ - จัดลำดับความสำคัญของเหตุการณ์

**คำแนะนำ**: กำหนดบริบทให้กับนักวิเคราะห์ว่าเหตุการณ์ใดควรมุ่งเน้นเป็นอันดับแรกโดยพิจารณาจากความรุนแรงของการแจ้งเตือนและความอ่อนไหวของสินทรัพย์ 

 
Microsoft Threat Protection ใช้การวิเคราะห์ความสัมพันธ์และรวบรวมการแจ้งเตือนและการตรวจสอบที่เกี่ยวข้องทั้งหมดจากผลิตภัณฑ์ต่าง ๆ ไว้ในเหตุการณ์เดียว Microsoft Threat Protection ยังทริกเกอร์การแจ้งเตือนที่ไม่ซ้ำกันเกี่ยวกับกิจกรรมที่สามารถระบุได้ว่าเป็นอันตรายเท่านั้นเนื่องจากความสามารถในการมองเห็นครบวงจรที่ Microsoft Threat Protection มีอยู่ทั่วทั้งอสังหาริมทรัพย์และชุดผลิตภัณฑ์ ด้วยการทำเช่นนั้น Microsoft Threat Protection จะบรรยายเรื่องราวการโจมตีที่กว้างขึ้น ช่วยให้นักวิเคราะห์การดำเนินงานด้านความปลอดภัยเข้าใจและจัดการกับภัยคุกคามที่ซับซ้อนทั่วทั้งองค์กร

- [จัดลำดับความสำคัญของเหตุการณ์ใน Microsoft Threat Protection](/microsoft-365/security/mtp/incident-queue?view=o365-worldwide&preserve-view=true)

### <a name="ir-6-containment-eradication-and-recovery--automate-the-incident-handling"></a>IR-6: การกักกัน การกำจัด และการกู้คืน - จัดการเหตุการณ์โดยอัตโนมัติ

**คำแนะนำ**: ทำให้งานแบบแมนวลที่เกิดขึ้นซ้ำเป็นแบบอัตโนมัติเพื่อเร่งเวลาตอบสนองและลดภาระให้กับนักวิเคราะห์ งานแบบแมนวลใช้เวลาในการดำเนินการนานกว่า ทำให้แต่ละเหตุการณ์ช้าลง และลดจำนวนเหตุการณ์ที่นักวิเคราะห์สามารถจัดการได้ นอกจากนี้งานแบบแมนวลยังเพิ่มความอ่อนเพลียของนักวิเคราะห์ ซึ่งจะเพิ่มความเสี่ยงต่อความผิดพลาดของมนุษย์ที่ทำให้เกิดความล่าช้า และลดความสามารถของนักวิเคราะห์ในการมุ่งเน้นไปที่งานที่ซับซ้อนให้มีประสิทธิภาพ  
 
ใช้คุณลักษณะการทำงานอัตโนมัติของเวิร์กโฟลว์ใน Microsoft Threat Protection เพื่อทริกเกอร์การตรวจสอบและการแก้ไขโดยอัตโนมัติตามการแจ้งเตือนความปลอดภัยที่เข้ามา 
 
- [การตรวจสอบและการตอบสนองอัตโนมัติใน Microsoft Threat Protection](/microsoft-365/security/mtp/mtp-autoir)

**การตรวจสอบศูนย์รักษาความปลอดภัย Azure**: ไม่สามารถใช้งานได้

### <a name="gs-1-define-asset-management-and-data-protection-strategy"></a>GS-1: กำหนดกลยุทธ์การจัดการสินทรัพย์และการป้องกันข้อมูล 

**คำแนะนำ**: ตรวจสอบให้แน่ใจว่าคุณจัดทำเอกสารและสื่อสารกลยุทธ์ที่ชัดเจนสำหรับการตรวจสอบและการปกป้องระบบและข้อมูลอย่างต่อเนื่อง จัดลำดับความสำคัญของการค้นหา การประเมิน การป้องกัน และการตรวจสอบข้อมูลและระบบที่สำคัญทางธุรกิจ 

กลยุทธ์นี้ควรมีคำแนะนำ นโยบาย และมาตรฐานสำหรับองค์ประกอบต่อไปนี้ในรูปแบบเอกสาร:

-   การอนุมัติองค์กรด้านความปลอดภัยของบริการ Azure สำหรับการใช้งาน 

-   การรักษาความปลอดภัยของสินทรัพย์ผ่านวงจรชีวิตของสินทรัพย์เหล่านั้น

-   กลยุทธ์การควบคุมการเข้าถึงที่จำเป็นตามการจำแนกประเภทข้อมูลขององค์กร

-   การใช้ความสามารถในการป้องกันข้อมูลของ Azure แบบเนทีฟและบุคคลที่สาม

-   ข้อกำหนดการเข้ารหัสข้อมูลสำหรับกรณีการใช้งานระหว่างารส่งผ่านและการพัก

-   มาตรฐานการเข้ารหัสที่เหมาะสม

**คำแนะนำ**: กำหนดกลยุทธ์ทั่วทั้งองค์กรเพื่อแบ่งเซกเมนต์การเข้าถึงสินทรัพย์โดยใช้การผสมผสานระหว่างข้อมูลเอกลักษณ์ เครือข่าย แอปพลิเคชัน การสมัครสมาชิก กลุ่มการจัดการ และการควบคุมอื่น ๆ

### <a name="gs-5-define-network-security-strategy"></a>GS-5: กำหนดกลยุทธ์ความปลอดภัยของเครือข่าย

**คำแนะนำ**: กำหนดแนวทางการรักษาความปลอดภัยเครือข่าย Azure ในฐานะที่เป็นส่วนหนึ่งของกลยุทธ์การควบคุมการเข้าถึงความปลอดภัยโดยรวมขององค์กรของคุณ  

กลยุทธ์นี้ควรมีคำแนะนำ นโยบาย และมาตรฐานสำหรับองค์ประกอบต่อไปนี้ในรูปแบบเอกสาร: 

-   การจัดการเครือข่ายแบบรวมศูนย์และความรับผิดชอบด้านความปลอดภัย

-   รูปแบบการแบ่งเซกเมนต์เครือข่ายเสมือนที่สอดคล้องกับกลยุทธ์การแบ่งเซกเมนต์ขององค์กร

-   กลยุทธ์การแก้ไขสำหรับสถานการณ์คุกคามและการโจมตีที่แตกต่างกัน

-   ขอบอินเทอร์เน็ตและกลยุทธ์การเข้าและออกของข้อมูล

-   ระบบคลาวด์แบบไฮบริดและกลยุทธ์การเชื่อมต่อระหว่างกันในองค์กร

-   สิ่งประดิษฐ์ด้านความปลอดภัยเครือข่ายที่ทันสมัย (เช่น แผนภาพเครือข่าย สถาปัตยกรรมเครือข่ายอ้างอิง)

### <a name="gs-6-define-identity-and-privileged-access-strategy"></a>GS-6: กำหนดข้อมูลเอกลักษณ์และกลยุทธ์การเข้าถึงที่มีสิทธิ์

**คำแนะนำ**: สร้างข้อมูลเอกลักษณ์ Azure และวิธีการเข้าถึงที่มีสิทธิ์ในฐานะที่เป็นส่วนหนึ่งของกลยุทธ์การควบคุมการเข้าถึงความปลอดภัยโดยรวมขององค์กรของคุณ  

กลยุทธ์นี้ควรมีคำแนะนำ นโยบาย และมาตรฐานสำหรับองค์ประกอบต่อไปนี้ในรูปแบบเอกสาร: 

-   ระบบข้อมูลเอกลักษณ์และการรับรองความถูกต้องแบบรวมศูนย์ และการเชื่อมต่อระหว่างกันกับระบบข้อมูลเอกลักษณ์ภายในและภายนอกอื่น ๆ

-   วิธีการรับรองความถูกต้องที่แข็งแกร่งในกรณีการใช้งานและเงื่อนไขที่แตกต่างกัน

-   การคุ้มครองผู้ใช้ที่มีสิทธิ์พิเศษสูง

-   การตรวจสอบและจัดการกิจกรรมของผู้ใช้ที่ผิดปกติ  

-   ข้อมูลเอกลักษณ์ของผู้ใช้และการตรวจสอบการเข้าถึงและกระบวนการสร้างความสอดคล้องต่อเนื่อง

### <a name="gs-7-define-logging-and-threat-response-strategy"></a>GS-7: กำหนดกลยุทธ์การบันทึกและการตอบสนองต่อภัยคุกคาม

**คำแนะนำ**: กำหนดกลยุทธ์การบันทึกและการตอบสนองต่อภัยคุกคามเพื่อตรวจจับและแก้ไขภัยคุกคามอย่างรวดเร็วในขณะที่ปฏิบัติตามข้อกำหนด จัดลำดับความสำคัญให้นักวิเคราะห์ด้วยการแจ้งเตือนที่มีคุณภาพสูงและประสบการณ์ที่ราบรื่นเพื่อให้พวกเขาสามารถมุ่งเน้นไปที่ภัยคุกคามมากกว่าการผสานรวมและขั้นตอนแบบแมนวล 

กลยุทธ์นี้ควรมีคำแนะนำ นโยบาย และมาตรฐานสำหรับองค์ประกอบต่อไปนี้ในรูปแบบเอกสาร: 

-   บทบาทและความรับผิดชอบขององค์กรปฏิบัติการด้านความปลอดภัย (SecOps) 

-   กระบวนการตอบสนองต่อเหตุการณ์ที่กำหนดไว้อย่างดีซึ่งสอดคล้องกับ NIST หรือกรอบงานอุตสาหกรรมอื่น 

-   การสกัดข้อมูลบันทึกและการเก็บรักษาบันทึกเพื่อรองรับการตรวจจับภัยคุกคาม การตอบสนองต่อเหตุการณ์ และความต้องการด้านการปฏิบัติตามกฎระเบียบ

-   ความสามารถในการมองเห็นและความสัมพันธ์ของข้อมูลภัยคุกคามแบบรวมศูนย์ โดยใช้ SIEM ความสามารถของ Azure แบบเนทีฟ และแหล่งข้อมูลอื่น ๆ 

-   แผนการสื่อสารและการแจ้งเตือนกับลูกค้า ซัพพลายเออร์ และบุคคลสาธารณะที่น่าสนใจ

-   การใช้ Azure แบบเนทีฟและแพลตฟอร์มของบุคคลที่สามในการจัดการเหตุการณ์ เช่น การบันทึกและการตรวจจับภัยคุกคาม 
การพิสูจน์หลักฐาน และการแก้ไขและกำจัดการโจมตี

-   กระบวนการจัดการเหตุการณ์และกิจกรรมที่เกิดขึ้นหลังเหตุการณ์ เช่น บทเรียนที่ได้เรียนรู้และการเก็บรักษาหลักฐาน
